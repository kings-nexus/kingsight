# Architecture Decision Records

> 重大架构决策的正式记录。防止同一个决策被反复重新讨论。

## 格式

```markdown
## ADR-N: [标题]
- **日期**: YYYY-MM-DD
- **状态**: 已决定 / 已废弃
- **上下文**: [为什么需要做这个决策]
- **决定**: [我们决定做什么]
- **后果**: [这个决定带来什么影响]
- **关联**: [相关 DP/DF/文件]
```

## 变更协议

ADR 是项目的"宪法"级文档。变更受 ADR-006（宪法保护机制）约束：
- **新增 ADR**: 需要人类提出 + architect agent 评估可行性和影响面
- **修订 ADR**: 需要人类审批 + architect agent 评估 + 明确记录修订理由和旧版本
- **废止 ADR**: 同修订，但更严格——需要说明废止后的替代方案
- **任何单方面修改（人类或 agent）均无效**，必须走评审流程

## ADR-001: 独立项目，不 fork gstack 或 ECC

- **日期**: 2026-03-18
- **状态**: 已决定
- **上下文**: 项目需要 gstack 的 agent team 架构和 ECC 的自进化思想，但有完整的产品路线图（教育系统 + prompt 研究 + prompt 写作 + continuous learning），方向与两个上游项目完全不同。Fork 会带来大量不相关代码的维护负担。
- **决定**: 创建独立项目。从 gstack 和 ECC 提取设计思想和模式，不创建运行时依赖。
- **后果**: 需要自建部分基础设施（模板系统、测试框架），但获得完全的架构自主权。成熟的功能可以反向贡献给上游。
- **关联**: README-prompt-system.md

## ADR-002: Agent team 强制用 subagent，不用 CLI subprocess

- **日期**: 2026-03-18
- **状态**: 已决定
- **上下文**: 项目中存在三种 AI 调用路径（subagent / CLI subprocess / 混合）。CLI subprocess 需要独立认证（API key 或 CLI 登录），而 subagent 继承当前会话认证，零配置。E2E 测试因依赖 `claude -p` 而要求 API key，但用户从来不需要 API key。
- **决定**: Agent team 开发和测试全部使用 subagent（Agent tool）。CLI subprocess 仅用于需要多模型或有状态会话的内容生产场景。项目代码永远不引用 API key。
- **后果**: 测试零配置，开发体验一致。但无法在 subagent 中调用非 Claude 模型（需要时走 CLI subprocess 的路径 2）。
- **关联**: subproject/ai_invocation_patterns_reference.md, DP-3

## ADR-003: 一 agent 一领域（修订自"一 agent 一任务"）

- **日期**: 2026-03-18（初版），2026-03-18（修订）
- **状态**: 已决定（修订版）
- **上下文**: 初版规定"一 agent 一任务"，将秘书（路由）和 prompt-writer（写作）设为两个独立角色。但分析发现秘书构造路由 prompt 本身就是最关键的 prompt writing——它高度场景化、和路由决策不可分离、需要完整的路由上下文。拆分会导致：(1) 每次路由多一跳 subagent 调用，(2) prompt-writer 需要复制秘书的全部路由知识，(3) 违反 DP-65（领域知识覆盖度相同时扩展现有角色优先于创建新角色）。
- **修订理由**: 判断标准从"任务"改为"领域"。秘书的领域是"用户意图理解 + 路由 prompt 构造"，这两个不可分；prompt-research 的领域是"技术验证 + 知识层级管理"，与秘书不重叠。
- **决定**: 每个 agent 角色覆盖一个内聚的领域，不按输出类型拆分。秘书负责路由 + 路由 prompt 写作（合并原 prompt-writer 的路由 prompt 职责）。prompt-research 负责通用 prompt 技术研究 + 模板注入（deploy 模式）。不再设独立的 prompt-writer agent。
- **后果**: 角色数量减少，秘书权限需要放宽（允许构造 prompt 文本，但仍不写入持久文件）。"帮我写一个通用 prompt"类请求由秘书分类为 B 类，路由给 prompt-research。
- **旧版本**: "每个 agent 做且只做一件事。Prompt researcher 只研究，prompt writer 只写，stack 只展示状态。"
- **关联**: DP-65, agent_secretary_reference.md

## ADR-004: 秘书 Agent ABCDL 五类消息分类协议

- **日期**: 2026-03-18
- **状态**: 已决定
- **上下文**: 秘书作为用户入口需要分类每条消息并路由给正确的 agent。分类基于两个正交轴：（1）谁来做——秘书自己/恢复流水线/新派发，（2）新派发的性质——探索/执行/学习。原参考设计 ABCDE 中的 E（个人决策）不适用于 prompt 工程工具，用 L（学习）替代，因为教育系统是本项目核心入口。
- **决定**: 采用 ABCDL 五类分类：A=探索未决定（→研究 agent），B=明确执行（→研究 agent），C=信息查询（→秘书直答），D=流程控制（→恢复流水线），L=学习请求（→教育系统）。模糊默认 A（偏向多想而非多做）。秘书只路由不执行，只读不写。
- **后果**: 每条用户消息有明确的分类路径。A/B 需要预览门控和质量清单，C/D 轻量处理，L 是 TDL 门控的反向入口（学习解锁能力）。分类基于用户意图性质而非目标 agent，目标 agent 是分类的结果。
- **关联**: ADR-003, agent_secretary_reference.md, DP-72/73/74

## ADR-005: 扁平化管理 — agent 和人之间不是专制上下级

- **日期**: 2026-03-18
- **状态**: 已决定
- **上下文**: 传统 AI assistant 设计中，人类下达指令、AI 执行，是单向的主从关系。但这个模型有根本缺陷：(1) 人类的 prompt 可能不完善，想法可能不正确——人类智商不一定高于 AI 在特定领域的判断力；(2) agent team 中 subagent 盲目服从主 agent 会导致错误放大——秘书路由质量差，下游 agent 全部产出废品；(3) "服从性太高"意味着系统丧失了纠错能力，所有错误都沿着指令链传播到底。
- **决定**:
  1. **人-agent 关系**: agent 有权质疑人类的指令。质疑方式是提供事实依据和替代方案，最终决策权仍在人类，但 agent 不是"收到指令立即执行"，而是"评估指令合理性 → 合理则执行 / 不合理则质疑"。
  2. **agent-agent 关系**: subagent 有权向主 agent 反馈"任务描述不够"、"这个方向可能有问题"。秘书的路由不是命令，是建议——目标 agent 可以要求补充信息或建议重新分类。
  3. **反馈机制**: 每个 agent 完成后，其结论可以触发上游修正（治理员质疑架构师、测试员驳回开发者），而非单向流水线。
  4. **限制**: 质疑必须基于事实和逻辑，不是情绪或偏好。质疑有上限（参考秘书文档 MAX_REVISION_ROUNDS = 1），超限升级到人类裁决。
- **后果**: 系统具备自纠错能力。但可能导致流水线推进变慢（多一轮质疑-回应）。通过设置质疑上限和自动升级来控制延迟。
- **关联**: ADR-006, agent_secretary_reference.md 第四节（设计质疑回退协议）

## ADR-006: 宪法保护机制 — 核心决策变更需要多方评审

- **日期**: 2026-03-18
- **状态**: 已决定
- **上下文**: ADR 和高级 DP 是项目的"宪法"——它们约束所有 agent 的行为。如果任何单一方（人类或 agent）可以单方面修改这些文档，宪法就失去了约束力。类比民主政府：总统不能单独修改宪法，需要参众两院评审通过。
- **决定**:
  1. **宪法级文档**: ADR.md, DP.md 中 L3 级别的原则
  2. **变更流程**:
     - 提案方（人类或 agent 均可）提出变更请求，说明理由
     - Architect agent 评估：变更的影响面、与现有 ADR/DP 的冲突、替代方案
     - 人类审查 architect 的评估，做出最终决定
     - 通过后，变更生效，旧版本保留在文档中（可追溯）
     - 未通过的变更记录为 DF（延迟项），附触发条件
  3. **紧急逃生阀**: 如果变更涉及安全问题或系统不可用，人类可以临时绕过评审，但必须在 24 小时内补充正式评审
  4. **不可单方面修改**: 无论是人类还是 agent，未经评审的 ADR/DP 修改均无效。agent 发现未经评审的修改应报告并回滚。
- **后果**: 核心决策稳定性提高，但变更速度降低。适合项目成熟期。早期阶段（当前）评审可以简化为"人类 + 一个 agent 评估"，不需要完整的多方评审。
- **关联**: ADR-005, docs/governance/DP.md, knowledge_governance_reference.md 第六节（ADR-006 宪法审查机制）
