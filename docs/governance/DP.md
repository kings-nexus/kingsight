# Design Principles

> 跨话题、跨版本有效的设计原则。从项目实践中积累，不预导入。

## 提升标准 (KC-1~KC-4)

每条"关键发现"满足以下**任意两条**即提升为 DP:

| 标准 | 名称 | 判断方法 |
|------|------|---------|
| **KC-1** | 跨话题有效性 | 换一个完全不同的场景，发现是否仍成立？ |
| **KC-2** | 跨版本有效性 | 在 >= 2 个不同版本/轮次中是否出现相似模式？ |
| **KC-3** | 方案导向性 | 可否用"遇到 X 类问题时，优先考虑 Y 而非 Z"来表述？ |
| **KC-4** | 反直觉性 | 一个不了解项目的人是否会做出相反的设计选择？ |

**分流**:
- 满足 >= 2 条 → 写入 DP
- 满足 1 条 → 保留在审计日志
- 满足 0 条 → 纯战术发现，不提升

## 格式

```markdown
### DP-N: [一句话原则名称]
- **来源**: [发现场景，日期]
- **精炼表述**: [可操作的一句话原则]
- **适用范围**: [何时引用]
- **反例/边界**: [何时不适用]
- **验证状态**: 未验证 / 部分验证([证据]) / 已验证([证据])
```

## 读写权限

| 操作 | 允许的角色 |
|------|-----------|
| 新增 DP | 架构师 / 研究员（需满足 KC 标准） |
| 读取引用 | 所有角色 |
| 更新验证状态 | 架构师（新证据出现时） |
| 废止 | 人类审批 |

## 预导入原则（从知识治理参考文档迁移，已验证）

### DP-3: 正面框架优于负面约束
- **来源**: 知识治理体系，多轮 agent 行为调优中发现
- **精炼表述**: 对 LLM 下达"不要做X"远不如"做Y"可靠。用 Bad→Good 示例锚定替代规则列举。
- **适用范围**: 编写 skill 模板、agent 指令、prompt 设计
- **反例/边界**: 绝对禁止项仍需负面约束作为兜底（如安全红线），但应同时提供正面替代路径
- **验证状态**: 已验证（原项目多轮实验 + 本项目 skill 模板编写实践）

### DP-63: 范例密度定律 — 一个完整好范例 > N 条精确规则
- **来源**: 知识治理体系，LLM 注意力机制对具体实例权重高于抽象规则
- **精炼表述**: 指引中规则:范例比应 <= 1:2（范例占主导）。
- **适用范围**: skill 模板编写、few-shot 设计、教育材料
- **反例/边界**: 纯逻辑约束（如 JSON schema）用规则更精确
- **验证状态**: 已验证（原项目多轮实验）

### DP-72: Advisory Gating + Outcome Tracking > Exam Gating + Input Evaluation
- **来源**: 知识治理体系，教育系统设计
- **精炼表述**: AI 系统不应通过考试预测人类能力。Advisory（信息展示+建议）+ outcome tracking（事后验证）优于 input evaluation。
- **适用范围**: 教育系统的门控设计
- **反例/边界**: 涉及安全风险的操作（如生产环境部署）可能需要更硬性的门控
- **验证状态**: 部分验证（原项目设计分析，未实施）

### DP-73: System-Centric Warnings > User-Centric Blame
- **来源**: 知识治理体系，人机交互设计
- **精炼表述**: 展示操作风险时归因于系统状态（"此模块有 3 次超时"）而非用户行为（"你上次失败了 4 次"）。
- **适用范围**: 教育系统反馈、错误提示、进度展示
- **反例/边界**: 用户主动请求自身表现数据时可以展示
- **验证状态**: 部分验证（原项目设计分析）

### DP-74: 预编写检查清单+用户自评 > AI 实时评估 ∨ 完全放弃门控
- **来源**: 知识治理体系，教育系统设计
- **精炼表述**: 中间路径：由专家预编写场景化清单，AI 展示清单，人类自评对照。考核的三重角色：门控 + 教学 + 诊断。
- **适用范围**: 教育系统的考核机制
- **反例/边界**: 可自动验证的技能（如 JSON 格式正确性）可用自动评估
- **验证状态**: 部分验证（原项目设计分析）

### DP-6: 单次调用单一焦点 — agent 一次只做一件内聚的事
- **来源**: prompt-research/secretary/stack 设计过程中反复出现的模式，2026-03-18
- **精炼表述**: 每次 agent 调用只处理一个模式或任务，模式在入口处确定，执行过程中不切换。避免精力分散导致质量下降。
- **适用范围**: 所有 agent 的 skill 模板设计。设计 agent 时必须在入口处做 mode detection，一个模式走到底。
- **反例/边界**: 秘书的 C 类（信息查询）本身就是轻量操作，不需要模式隔离。流水线中的"汇报+启动下一个"是两个原子操作的顺序执行，不是混合模式。
- **验证状态**: 部分验证（prompt-research 四模式设计有效，secretary ABCDL 分类有效）
- **当前保证机制**: 仅靠 skill 模板中的 mode detection 分支（文字规范）。无代码级强制。
- **升级路径**: 当出现 agent 在一次调用中混合多个模式导致质量问题时（DP-62 触发条件：连续 >= 3 次同类问题），将 mode 隔离从 prompt 层迁移到代码层（hook 或 wrapper 强制单模式执行）。记录为 DF 监控。

### DP-7: 分析根因不是交付物，修复机制才是
- **来源**: 本项目多次遗漏（commit/README/E2E/symlink），每次只做"分析为什么忘了"但不设计防遗漏机制，导致反复犯同类错误，2026-03-18
- **精炼表述**: 遇到问题时，交付物不是"为什么出了问题"（根因分析），而是"怎么让这个问题不再发生"（修复机制）。根因分析是手段，不是目的。Agent 尤其容易停在"分析完毕"就结束，因为分析本身消耗了大量 token 给人一种"做了很多事"的错觉。
- **适用范围**: 所有 agent 的问题响应模式。任何错误修复流程的交付物必须包含一个结构性变更（代码/配置/清单），不能只是"记住下次注意"。
- **反例/边界**: 探索性研究（prompt-research scout 模式）的交付物确实是分析和假设，不需要修复机制。此原则适用于**已知问题的修复**，不适用于**未知领域的探索**。
- **验证状态**: 已验证（本 session 内：遗漏 commit → 加了 memory 但继续遗漏 → 加了 CLAUDE.md Definition of Done 清单 = 结构性修复）
