## LinuxDo 帖子（中等深度，技术社区，引发讨论）

### 帖子 1 — 控制感焦虑（"我失去了对项目的控制"）

**标题：用了三个月 AI 编程，我发现自己变成了项目里最弱的一环**

说出来有点丢人，但我觉得这个圈子里可能不止我一个人有这种感觉。

我用 Claude Code 搞了三个月的项目。agent team 越来越强，代码质量越来越高，测试覆盖越来越全。但有一天我突然发现——我已经不理解自己的项目在做什么了。

不是说完全看不懂代码。是那种更微妙的脱节：agent 提交了一个架构提案，我看了标题觉得有道理，按了回车。agent 又提交了一个 PRD，我扫了一眼，按了回车。测试全过，发布成功。

然后有人问我"你这个 ABCDL 分类是怎么设计的"，我答不上来。那是我自己项目的核心功能。

后来我想明白了：这不是懒的问题，是一个结构性陷阱。**AI 越强，你越不需要理解就能推进。你越不理解，你的指导越差。你的指导越差，AI 越绕着你走。** 这是个正反馈循环，会自我加速。

所以我做了一个教育系统——不是写文档让自己看（没人看文档），而是在我要做某个操作之前，先花 8 分钟做 3 个场景测试。考试就是课程。做错了系统告诉你为什么错，做对了就不再提醒。

整套系统开源了：https://github.com/kings-nexus/kingsight

有人也有这种"我不知道 agent 刚才做了什么但测试通过了"的感觉吗？

### 帖子 2 — 被替代焦虑（"AI 让我变成了按回车的人"）

**标题：你有没有发现——你写给 AI 的 prompt 才是整个系统最烂的部分**

我最近在做一个 agent team 项目，6 个 AI agent 组成流水线：架构师提方案 → 治理员审查 → 研究员写规格 → 开发者实施 → 测试员验证。

每个 agent 的输出质量都很稳定。但整个系统的瓶颈在哪？

在我。

我写给秘书 agent 的那句话——"我觉得应该优化一下路由"——这句话的信息密度是整条流水线里最低的。agent 收到这种空洞的指令，能产出什么？空洞的分析。

所以我给秘书加了一个 6 维度质量检查：数据点够不够？有没有根因链？有没有场景对比？文件路径精确吗？用户原话保留了吗？引用了已有的决策记录吗？不够 4/6 就不让路由，先让秘书自己补充调查。

结果发现：**检查 prompt 质量之后，整条流水线的输出质量直接翻倍。** 因为在串联架构里，上游差一分下游差十分。

这个项目叫 kingsight（https://github.com/kings-nexus/kingsight），核心想法是：**不是 AI 不够强，是你给 AI 的输入不够好。** 与其花时间调 agent 的 prompt，不如花时间提高自己给 agent 的指令质量。

### 帖子 3 — 技术好奇（"缓存系统"框架，硬核角度）

**标题：从 Raw Prompt 到 Secretary Agent——AI 编程的 6 层缓存架构**

我整理了一个框架，把 AI 编程工具的演进看成缓存系统的层级叠加：

```
Layer 0: Raw Prompt（无缓存，一句句推）
Layer 1: Skill（缓存单任务执行路径）
Layer 2: Pipeline（缓存任务间依赖）
Layer 3: Agent（缓存运行时决策）
Layer 4: Agent Team（缓存专业分工）
Layer 5: Secretary Agent（缓存用户意图理解）
```

每往上一层，你固化的决策维度越高，需要亲自介入的频率越低。

现在大部分项目停在 Layer 4——GStack、agency-agents、OpenCrew 都是很强的 agent team，但它们假设你知道该找谁、怎么说。你还是自己的调度中心。

Layer 5 是在 agent team 上面加一个秘书：你说一句模糊的话，秘书搞清楚你到底在做什么（用的是 ABCDL 五类意图分类，区分的不是"你在说什么领域"而是"你在做什么动作"），然后构造高质量的任务简报交给团队。

完整分析：https://github.com/kings-nexus/kingsight/blob/main/docs/article-cache-system.md
项目：https://github.com/kings-nexus/kingsight

你觉得还有 Layer 6 吗？

### 帖子 4 — 方法论好奇（"agent 能对人说不"）

**标题：我的 AI agent 拒绝了我的指令，然后我意识到这才是对的**

我给秘书 agent 写了一条规则：**agent 可以质疑人类的指令。**

听起来很疯狂？但是想想看——

我让秘书跳过预览门控"就这一次"。秘书说："预览门控是强制步骤。跳过可能导致路由错误。确定要跳过吗？"

我让秘书直接修改架构文件。秘书说："我的权限不包括修改文件。应该路由给开发 agent。"

我让 agent team 跳过治理员审查直接开发。系统说："这个提案还没有经过冲突检测和依赖分析。"

每一次被拒绝都让我不爽。但每一次它都是对的。

这个项目叫 kingsight（https://github.com/kings-nexus/kingsight）。它有一条架构决策（ADR-005）：**agent 和人之间不是上下级关系，是扁平协作。** 质疑必须基于事实和逻辑，有轮次上限，超限升级到人类裁决。但 agent 不是你的下属——它是你的同事。

而且它还有一条宪法保护（ADR-006）：**核心决策不能被任何单方面修改——包括人类。** 要改宪法得走评审流程，就像修宪需要参众两院投票一样。

你能接受你的 AI 对你说"不"吗？

### 帖子 5 — 意义焦虑（"人类还有存在价值吗"）

**标题：AI agent 越来越强，人类的角色到底是什么？我做了一个实验**

在做这个项目的过程中，我和 AI 做了一场辩论：

**反方（AI 提出）**：随着 AI 发展，人类心智带宽迟早跟不上 agent。让智力更低的人类来指导智力更高的 agent 本身就不合理。应该把精力花在设计自运行系统上，而不是试图教人类跟上硅基生命。

**正方（我的回应）**：当前 AI 不能可靠自递归强化。本 session 中 agent 反复犯同类错误——遗漏 commit 3 次、违反自身黑名单 5 次、推荐错误的下一步任务 2 次。每次都需要人类纠正。

最后的结论让我不太舒服但接受了：**教育系统是一个被设计成让自己变得不必要的系统。** 就像脚手架——建筑完成后拆除。先教人理解系统，等 AI 可靠到不需要人理解了，再拆掉教育系统。

但"什么时候 AI 够可靠"这个问题，必须由人来判断——不能让 AI 自己说"我已经够好了你可以走了"。

项目：https://github.com/kings-nexus/kingsight

这个问题没有标准答案。你怎么看？
