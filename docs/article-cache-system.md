# 你一直在建缓存系统，只是你没意识到

不知道大家在使用 Claude Code 的时候有没有发现——我们其实一直在做一件事：建立缓存系统。

只不过我们缓存的不是数据，而是人类决策。

每往上一层，你固化的决策维度越高，需要亲自介入的频率越低。从手动 prompt，到 skill，到 pipeline，到 agent，到 agent team——这条路你可能已经走了大半。但我发现，这条路的尽头还差一层。

这篇文章把这套演进梳理清楚，然后介绍我正在做的最后一层：Secretary Agent。

---

## Layer 0 — Raw Prompt（无缓存）

最开始的时候，所有人都是手动推动任务。一句一句 prompt 驱动，没有任何复用。

你就是那个 for 循环里的调度器，每一轮都得亲自到场。

你让 Claude Code 帮你写一个 React 组件，得先说"创建文件"，再说"加一个 state"，再说"写个 useEffect"，再说"加上错误处理"……每一步都要你来推。做完一个组件，下一个组件又是同样的流程。

这种工作方式能用，但不可持续。你会发现很多对话其实在重复同样的模式。

## Layer 1 — Skill（缓存：单任务执行路径）

后来有人发现，一个固定模式的任务需要人一轮轮 prompt 推动，太重复，太无聊了。

于是把"需要多轮 prompt 才能跑完的一个固定任务"封装成一次调用——这就是 skill，第一层缓存。

拿 Claude Code 来说，skill 就是一个 SKILL.md 文件，里面写清楚了任务的步骤、需要的工具、输出格式。你说一句"用 docx skill 帮我生成报告"，它就按预设路径跑完了，不需要你一步步盯着。Anthropic 在 2025 年 10 月正式推出了这套 skill 体系，现在社区已经有几百个 skill 在流通——从 PDF 生成到 Playwright 测试到代码审查，各种场景都有人封装好了。

**缓存的是：某件具体的事怎么做。**

但一个大任务不可能全塞进一个 skill。有些模块能拆出来做成小 skill，插件式调用，大家像搭积木一样拼出一个大任务。问题是：不是所有任务都能干净地拆成几个原子 skill，而且手动编排它们的顺序和数据传递也很麻烦。

于是有了 pipeline——Dify、n8n 最早就是解决这个问题的。

## Layer 2 — Pipeline（缓存：任务间依赖关系）

Pipeline 是多个 skill 的组合，按 DAG（有向无环图）结构处理有先后依赖的复杂任务流。

一个大任务总是不能被完美拆分为完全独立的子任务——子任务之间有依赖、有状态传递。

举个具体的例子：你要做一份"竞品分析报告"。先爬取数据，再清洗格式，再做对比分析，再生成图表，最后排版成文档。每一步的输出是下一步的输入，跳过任何一步都不行，顺序也不能乱。这正是 pipeline 存在的原因。

Skill 解决"怎么做"，pipeline 解决"按什么顺序做、谁的输出是谁的输入"。Dify 和 n8n 就是典型的可视化 pipeline 搭建工具——拖拽节点，连线定义数据流。

**缓存的是：任务和任务之间的编排逻辑。**

但 pipeline 是静态的——你得事先把所有分支都设计好。现实中很多任务，下一步该做什么取决于上一步的结果，你没办法提前穷举所有路径。每次都要人来判断"现在该走哪条路"，这又回到了手动推动的老问题。

## Layer 3 — Agent（缓存：运行时决策）

Agent 解决的就是这个问题：把"遇到什么情况做什么判断"写进规则文件。

拿 Claude Code 举例，就是你的 CLAUDE.md——里面写清楚：遇到什么情况调哪个 skill，走哪条 pipeline，满足什么条件算任务完成，什么时候应该停下来问人。Agent 读这份规则，在运行时自己做决策，人不用再盯着。

这也是 Anthropic 在《Building Effective Agents》里强调的核心区分：workflow 是预定义代码路径编排 LLM 和工具（也就是 pipeline），而 agent 是 LLM 动态指导自己的流程和工具使用。Pipeline 是你画好的地图，agent 是一个能看路况自己导航的司机。

Agent 的判断力不是魔法——它来自你写好的规则和模型自身的推理能力。

**缓存的是：在运行时选择工具和路径的判断能力。**

但单个 agent 的规则文件能承载的东西是有限的。当任务横跨多个领域——既要调研又要执行又要写文档——一个 agent 塞太多职责就开始犯错，规则之间互相冲突，上下文也扛不住。你还是要人来拆分"这部分让谁做"。

## Layer 4 — Agent Team（缓存：专业分工）

自然地，你会想拆出多个专职 agent：一个负责研究，一个负责执行，一个负责教学，各自有自己的规则文件，只做自己最擅长的事。

这在 2026 年的 Claude Code 生态里已经是主流模式了。

GStack（Y Combinator CEO Garry Tan 开源的项目，16000+ star）把 Claude Code 变成了一支虚拟工程团队——CEO 审产品方向（/plan-ceo-review）、工程经理锁架构（/plan-eng-review）、QA 开真实浏览器实测（/qa）、发布工程师提 PR（/ship）。Garry Tan 自己用这套流程一周平均产出 10000 行代码和 100 个 PR，同时跑 10-15 个并行 sprint。

agency-agents 更激进，直接打包了一整个"AI 代理机构"——从前端专家到安全审计到 Reddit 社区运营，每个 agent 都有自己的人格、流程和交付标准，还支持跨工具兼容（Claude Code、Copilot、Gemini CLI、Cursor 都能用）。

OpenCrew 把 Slack 频道当岗位，每个频道绑一个专职 agent（CoS 幕僚长、CTO、Builder、KO 知识官……），thread 就是工单。它的设计者是一个经济学/MBA 背景的非技术用户——他甚至不用写代码，通过 Slack 就管理着一支 7 人 AI 团队。

**缓存的是：领域分工与协作结构。**

看起来很完美？问题来了。

**你仍然是调度员。** 在 GStack 里，你得自己决定现在该敲 /plan-ceo-review 还是 /qa 还是 /ship。在 agency-agents 里，你得说"Use the code-reviewer to check my changes"——你得知道有这个 agent、它叫什么、什么时候该叫它。在 OpenCrew 里，你得知道该去哪个频道 @哪个 bot。

团队的能力已经很强了，但使用团队的认知负担全在你身上。每次想完成一件事，都得先在脑子里过一遍"这事归谁管"，然后切到对应的 agent，用对应的方式下指令。

你变成了自己的调度中心——这不是又回到 Layer 0 的状态吗？只不过你调度的对象从 prompt 变成了 agent。

## Layer 5 — Secretary Agent（缓存：用户意图理解）

所以你需要一个秘书。

就像老板不需要记住公司每个员工的职责、沟通风格和工作习惯——秘书替他记着这些。你只需要告诉秘书你想做什么，秘书知道该找谁、怎么跟他们说。

**这一层目前没有人在做。**

我搜遍了 2026 年 3 月 star-history 上 Coding AI 排行榜前 20 的项目——OpenClaw（+6k）、agency-agents（+5.8k）、GStack（+3.6k）、Ruflo、agent-teams-lite、Overstory——没有一个项目在做这一层。 它们都停在 Layer 4：提供了越来越强的 agent team，但假设用户知道该找谁、怎么说。

最接近的有两个方向：

一是 **OpenAI Swarm 的 triage agent 模式**。 它会对用户消息做意图分类后路由给专家 agent——这个思路是对的。但它分类的维度是领域：这是 billing 问题还是 refund 问题？"我觉得退款流程可能有问题"和"退款"在它看来都是 refund 领域，都会被路由给退款 agent。

二是**学术界最近的 AgentOS 论文**。 它提出了 Agent Kernel 的概念——用一个统一的自然语言入口（叫 Single Port）替代传统桌面，内部做意图解析后分发给不同的 Skill Module。架构思路和我的 Secretary 几乎同构，但它还在论文阶段，没有解决意图模糊性这个核心问题。

### ABCDL：区分的不是"你在说什么"，而是"你在做什么"

Secretary Agent 做的第一件事，不是执行，而是分类你的意图。

这就是 ABCDL 分类系统：

| 类 | 你在做什么 | 秘书怎么做 | 例子 |
|----|----------|----------|------|
| A | 思考、探索、还没决定 | 路由给研究 agent | "我觉得…"、"要不要试…" |
| B | 明确要执行某件事 | 路由给执行 agent | "跑实验"、"部署 F-003" |
| C | 问一个事实问题 | 秘书直接回答 | "有多少个 ADR？" |
| D | 接着上次继续 | 恢复流水线状态 | "继续"、"选A"、"批准" |
| L | 想学习某个概念 | 路由给教育系统 | "教我…"、"不理解…" |

和 Swarm 的 triage agent 相比，关键区别在于：**同一个话题可以是 A 也可以是 B。** "我觉得应该优化一下部署流程"和"优化部署流程"——领域完全相同，但系统应该做的事情完全不同。前者应该去研究 agent 讨论可行性，后者应该直接去执行 agent 开工。这个区分在现有的所有 triage/router 系统中都不存在。

**为什么不能少于 5 类？** 因为每对合并都会出问题。A+B 合并，"我觉得应该…"就会被当成执行指令——团队执行力越强，这种误判的代价越大。C+D 合并，"继续"会被当成新查询。

**模糊时默认 A：宁可多想，不要多做。** 过度思考可以纠正，过早执行可能不可逆。

**缓存的是：把"用户说了什么"翻译成"系统该做什么"这个理解过程本身。**

### 但秘书不只是分类器

如果秘书只是把你的消息分个类然后转发，它就只是 Swarm 的高级版本。秘书真正不一样的地方在于：**它会在转发之前自己做功课。**

你说"我觉得 architect 的 Phase 3 有问题"。一个路由器会把这句话直接丢给 architect。但你回忆一下现实中的场景——你跟老板的秘书说"帮我约一下张总"，好的秘书不会直接打电话过去，她会先查张总最近的日程、确认上次会面遗留的问题、准备好相关资料。

我的秘书做的是同样的事：读文件、查历史、翻治理文档，然后把这些事实数据连同你的原话一起打包，构造一份密度远高于你那句话的任务简报。

然后——**在派发之前先给你看一遍**。

为什么？因为你说了一句话，但你没有说任务简报。秘书把你的意图翻译成了系统能理解的格式——**这个翻译过程是误传最可能发生的地方。** 让你过一眼，误传就消失了。

这里有一个不太舒服的事实：在串联架构中，**上游差一分，下游差十分。** 秘书构造的简报质量直接决定了 architect 的分析质量，architect 的分析质量决定了治理员的审查质量。一路传下去，秘书偷懒或者翻译错了的那一点点偏差，会被每一层放大。

所以秘书不是团队里最简单的角色。它是最关键的。

### 然后我发现了一个更尴尬的问题

做完 Secretary，我以为事情结束了。Layer 0 到 Layer 5，缓存系统完工。

然后我回头看了一下自己在开发过程中干了什么。

我在这个项目中批准了我没看懂的架构提案。我对着 agent 的输出按了回车，因为我不知道怎么评价它是好是坏。我指导 agent "去优化秘书的路由"——但我自己说不清楚路由系统是怎么工作的。agent 很配合地去执行了，结果方向是错的，我们花了很多时间返工。

我不是一个懒惰的用户。我是一个**没有能力判断 agent 产出的用户**。这两件事看起来很像——都是按回车——但根因完全不同。前者是态度问题，后者是结构问题。

**agent 越强，你越不需要理解就能推进项目。你越不理解，你的指导越差。你的指导越差，agent 越绕着你走。这是一个正反馈循环，会自我加速。**

打破点在哪？不是让 agent 变弱（那是倒退），是让人变强。不是写文档让人去读（没人读文档），是在人需要理解的那个瞬间，用最高效的方式让他理解。

所以我在 Secretary 里加了一层教育系统。

### 考试就是课程

当你发一条 A 类消息，秘书会先检查：你对相关知识域的理解够不够？如果够——你什么都感觉不到，消息正常路由。如果不够：

```
"这个操作需要了解 ABCDL 分类。你当前未评估过。
  A) 现在学（~8 分钟）
  B) 跳过继续
  C) 30 秒速览"
```

选 A 之后，你不会看到一篇文档。你会直接遇到 3 个场景：

> *"用户发送：'我觉得应该重新设计 Phase 3'——分类为什么？为什么不是 B？"*

你回答。然后系统揭示正确答案、推理链、以及你的答案为什么错。**这个过程本身就是教学。** 不是先教再考，是先考再教——TDD 的思路：先写一个会失败的测试，然后只学够让测试通过的知识。

教育学有个概念叫 testing effect（检索练习效应）——测试本身比重复阅读更能促进长期记忆。这不是我发明的，是认知科学的基本结论。我只是把它工程化了。

最后一步是自评——不是系统判断你过没过，是你自己勾选"我觉得我理解了这些概念"。系统尊重你的判断（永远不阻断你），但它会记住你的诊断分数。如果你做错了 3 题但勾了"全都理解"——下次提醒频率更高。

有人会问：这不会被投机取巧绕过吗？

会的。但每条投机取巧的路径都通向学习：提前看考题？看完就等于学了。全部跳过？系统记录，下次更频繁提醒。自评造假？诊断分数独立记录，advisory 会自动调整。

**这个系统不可能被 game 到"你没学到任何东西"的结果。** 这是设计出来的，不是偶然的。

## 总结

每一层缓存，都在把原本需要人脑实时决策的东西，固化成系统可以自动复用的结构：

| 层级 | 缓存了什么 | 人不再需要… |
|------|----------|-----------|
| Skill | 动作 | 一步步 prompt 推动 |
| Pipeline | 顺序 | 手动编排 skill 的先后 |
| Agent | 判断 | 在运行时决定走哪条路 |
| Agent Team | 分工 | 决定让谁来做 |
| Secretary | 意图 | 决定自己在做什么 |
| + 教育系统 | 理解 | 担心自己跟不上项目 |

从 Layer 0 到 Layer 5，你作为人类介入的粒度从"每句话"变成了"说一句话"。这句话可以是模糊的、随意的、甚至是自言自语——Secretary 会搞清楚你到底在做什么，然后把它变成系统能执行的东西。

而教育系统确保你说的这句话是有质量的。不是靠限制你能说什么，而是靠让你理解系统在做什么——理解之后，你自然会说出更好的话。

**这是最后一层缓存，因为它缓存的是你和系统之间的翻译层。在这一层之上，你就只是一个有想法的人，而不再是一个操作员。**

## 关于这篇文章和这个项目

诚实声明：这篇文章和它描述的整个系统，都是人类和 AI agent 协作完成的。

具体来说：Layer 0-4 的框架梳理和 ABCDL 的核心洞察是我（人类）写的。Secretary 和教育系统的详细设计是我和 Claude Opus 4.6 一起做的——我提出方向和约束，它做调研和方案设计，我审查和修改，它实现和测试。这篇文章的后半部分也是 AI 起草、我修改的。

这不是谦虚——这就是这个系统本身想要解决的问题的最好例证。**我没有能力独自设计一个 6 agent 的协作系统。** 但我有能力判断设计方案的好坏（大部分时候），有能力提出 AI 想不到的约束（比如"agent 应该能对人说不"），有能力在 AI 犯错时纠正方向（本文的教育系统就是因为 AI 反复在同一类错误上栽跟头才做的）。

人类和 AI 在这个项目中的关系不是"我指挥它干活"，更接近"我们各自做自己擅长的部分"。而教育系统存在的原因，是为了确保"我擅长的部分"不会随着项目变复杂而缩水到零。
