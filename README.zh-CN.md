# 你根本不理解你的 AI 助手在做什么

你用 AI 写代码已经好几个月了。你批准了你没看的提案。你对着你看不懂的输出按了回车。你知道代码能跑，因为测试通过了——但如果有人问你"agent 为什么做了这个设计决策"，你答不上来。

**这不是你懒。这是一个结构性陷阱。** AI 越强，你越不需要理解就能推进项目。你越不理解，你的指导就越差。你的指导越差，agent 就越绕着你走。直到有一天你发现——你不是在指导项目，你只是在按回车。

我们做了一个系统来打破这个循环。

## 你打开项目后会发生什么

```
你: "我觉得应该重新设计 architect 的 Phase 3"

系统: 在把这条消息路由给 architect 之前，我注意到你还没有
  了解过 team 流水线的工作方式。

  这不是一个你会"挂掉"的考试——是 8 分钟的真实场景，
  会让你看清系统实际上是怎么运转的。随时可以跳过。

  A) 现在学（~8 分钟）
  B) 跳过，直接继续
  C) 30 秒速览
```

如果你选 A，你会遇到三个真实场景——不是背定义，而是你真的会碰到的情况：

> *"用户发送：'我觉得应该重新设计 Phase 3，增加第三个攻击模型'——这条消息应该怎么分类？"*

你回答。然后系统告诉你实际会发生什么，以及为什么。8 分钟后你理解了一个靠自己摸索可能需要几周才能搞明白的东西。

之后，提示消失了。你看过了，你懂了，系统让开。

**考试就是课程。** 我们偷师了 TDD 的思路——先写一个会失败的测试，然后只学刚好够让测试通过的东西。大多数人从"看到自己的错误答案被解释"中学到的，比读文档多得多。

## 里面有什么

六个 AI agent，各司其职，组成流水线：

```
你的消息 → 秘书（分类 + 路由）
                ↓
  架构师 → 治理员 → 研究员 → 开发者 → 测试员 → 提交
```

架构师发现问题、提出方案，然后**用第二个 AI 模型（Gemini）攻击自己的方案**来找漏洞。只有经过跨模型对抗审查存活下来的提案才能通过。治理员可以质疑架构师的提案，把它打回去修改。测试员不只是跑测试——它会检查实现是否真的符合架构师最初的设计意图（设计漂移检测）。

**这些 agent 会互相吵架。** 也会跟你吵。告诉秘书跳过预览门控，它会怼回来：*"预览门控是强制步骤。跳过可能导致路由错误。确定要跳过吗？"* 你可以强制执行——你始终可以——但系统会确保你知道自己在做什么。

这不是一个听话的 AI 助手。这是一个协作的团队。

## 为什么这件事重要

今天所有的 AI 编程工具都有同一个问题：人类在不知不觉中变成了最短板。你是一条越来越强的链条中最弱的一环，但没有人告诉你。

我们选了一条不同的路：

**教人。** 不是用文档（没人看文档）。不是用教程（没人看完教程）。而是用场景——让你在 8 分钟内产生"哦，原来是这么回事"的感觉。然后让开，让你干活。

**让 agent 能说不。** 当你的指令和已有的架构决策冲突时，系统会告诉你——用证据，不是用权威。你可以 override，但 override 会被记录，系统会从中学习。

**让每条作弊路径都通向学习。** 想提前看所有考题？随便看——看完就等于学了。想自评"我全都懂了"但实际上做错了 3 题？系统会把你的诊断分数单独记录，自动调整提醒频率。这个系统的每一种"投机取巧"最终都会让你学到东西。

## 教育系统

4 个核心知识域（"Golden Rules"——不理解这些真的会帮倒忙）：

1. **消息分类** — 你说的话是怎么被路由到正确 agent 的
2. **质量门控** — 系统为什么要在行动前先给你看它的计划
3. **宪法保护** — 设计决策是怎么被保护起来不被误改的
4. **团队流水线** — 哪个 agent 干什么，为什么它们不越界

12 个场景测试，每一个都来自这个项目真实开发过程中遇到的情况。

## Agent 团队

| Agent | 一句话 |
|-------|--------|
| **秘书** | 路由你的消息，给其他 agent 构造高质量 prompt，跑教育门控 |
| **架构师** | 发现问题，提出方案，用第二个 AI 模型攻击自己的方案 |
| **治理员** | 审查提案的冲突和依赖，可以质疑并打回 |
| **研究员** | 把批准的提案翻译成精确的实施规格 |
| **开发者** | 忠实地按规格实施，偏差必须记录 |
| **测试员** | 验证实现是否符合设计意图，检测跨文件的语义不一致 |

## Prompt 研究

一个专门的 agent 通过受控实验发现哪些 prompt 技术真正有效——不是靠博客文章，是靠跨模型验证（Claude + Gemini）。知识分级：观察必须经过多实验+多标准验证才能升级为原则。

## 治理

9 个架构决策（ADR）——修改需要多方评审，你自己也不能单方面改。16 条设计原则，来自真实的错误。11 个延迟项，等着触发条件满足后复活。系统有宪法，没有人凌驾其上。

## 不舒服的真相

这个系统诞生于一个人的反复失败——它的创造者反复批准自己看不懂的 AI 输出，给出降低项目质量的指令，最终失去了对一个本该由他主导的项目的控制。

每一个功能都是因为先出了问题才存在的。教育系统存在，是因为创造者无法解释自己的 agent 在做什么。预览门控存在，是因为秘书一直在跳过人类审查直接派发任务。宪法保护存在，是因为决策被反复误改。

如果你在用 AI 编程工具，但从来没有那种"我不知道刚才发生了什么但测试通过了"的心虚感——这个项目不适合你。

如果你有过——欢迎。你不是一个人，这也不是你的错。工具的设计就是让"理解"变成可选项。我们设计了一个系统，让"理解"变成默认。

## 试一下

```bash
git clone https://github.com/kings-nexus/kingsight.git ~/.claude/skills/kingsight
cd ~/.claude/skills/gstack && ./setup
```

开始一段对话。发任何消息。系统会分类你的消息，检查你是否了解相关知识域，然后要么教你，要么路由你的请求。

前 30 分钟会改变你对"和 AI agent 一起工作"这件事的理解。
